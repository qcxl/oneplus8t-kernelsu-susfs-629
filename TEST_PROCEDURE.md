# 测试流程与移植规范

> 本文档分为两层：
> - **第 I 部分：通用移植方法论**——适用于所有内核功能移植（SUSFS、Magisk、KPM 等）
> - **第 II 部分：项目上下文**——当前项目的特定参数，每次新项目替换此处即可

---

# 第 I 部分：通用移植方法论

## ⚠️ 移植前强制检查清单（每次移植必须逐项完成，不得跳过）

### 1. 源码阅读（规范化流程）

> **阅读顺序**：头文件 → 函数声明 → 结构体定义 → 函数实现 → VFS 调用点 → 补丁文件

#### 1a. 准备：确定源文件版本
- [ ] 确认**基线版本**的实际来源（gitlab 原始版？GitHub 镜像版？哪个 commit/tag？）
- [ ] 确认**目标版本**的实际来源
- [ ] ⚠️ **关键风险：本地下载的镜像版本可能与 CI 构建用的版本不同**。例如 hypermezo4 镜像（v1.5.9）与 gitlab 原始版（可能 v1.5.5）存在 struct 差异。以 CI 实际使用的源为准

#### 1b. 结构性阅读（按顺序）
- [ ] **头文件 → 枚举/常量**：完整读取 `susfs_def.h`，记录所有 `CMD_*`、`INODE_STATE_*`、`AS_FLAGS_*`、`TIF_*`、`SUSFS_*_SIZE` 等常量
- [ ] **头文件 → 结构体**：完整读取 `susfs.h`，记录所有 `struct st_susfs_*` 的字段列表和顺序。**基线版本和目标版本逐字段对比**，标记新增/删除/改名的字段
- [ ] **头文件 → 函数声明**：完整读取函数签名，逐函数签名对比两版本的变化：
     - 返回类型是否变化（`int` → `void`）
     - 参数是否变化（单指针 → 双指针）
     - 参数类型是否变化（`ino` → `inode*`）
- [ ] **函数实现**：读取目标版本函数体的完整代码，标注每个外部依赖（调用了哪些宏、哪些外部函数、哪些全局变量）
- [ ] **补丁文件**：读取 `50_add` 补丁或其他集成补丁，确认 VFS 挂钩的精确插入位置和上下文

#### 1c. 依赖追踪（每个新功能必须独立完成）
- [ ] 全局变量：列出所有新引入的 `extern` 变量，逐一确认在目标内核中是否存在（搜索范围包括 KSUN 源码、内核头文件、其他模块）
- [ ] 锁机制：标注函数使用的锁类型（`spin_lock`、`mutex`、`SRCU`、`RCU`），确认目标内核版本支持且对应头文件已包含
- [ ] 外部函数：列出所有调用的函数（`kern_path`、`kzalloc`、`override_creds`、`get_fuse_inode` 等），逐一确认在目标内核中可用
- [ ] 宏/内联函数：列出所有使用的宏（`WRITE_ONCE`、`set_bit`、`test_bit`、`hash_for_each_possible_rcu` 等），确认在目标内核版本中定义

#### 1d. 经验与常见陷阱
| # | 经验 | 来源 |
|---|------|------|
| 1 | **本地文件 ≠ CI 源文件**。GitHub 镜像可能比 gitlab 原始版新（如 v1.5.9 vs v1.5.5）。struct 定义、函数签名都可能不同 | E002/E005/E009 |
| 2 | **结构体必须逐字段对比**。两版本结构体可能字段不同（v1.5.5 的 `st_susfs_open_redirect_hlist` 只有 4 字段，v2.2.0 有 9 字段）。不能假设结构体兼容 | Batch 3 预读 |
| 3 | **函数签名变化必须映射**。`int func(struct xxx* __user)` → `void func(void __user **)` 变化涉及全部参数的传递方式 | E001-E003 |
| 4 | **VFS 挂钩点从 50_add 补丁中找**。不是从目标版本代码中猜测。补丁中的行号和上下文是精确的 | Batch 2 经验 |
| 5 | **读头文件时注意 `#ifdef` 嵌套**。结构体可能在不同 `#ifdef` 条件中定义，同一个结构体名可能在不同条件下有不同定义 | E005 |
| 6 | **不要假设"这个函数肯定存在"**。如 `ksu_cred` 在 KSUN 中存在但路径与预期不同，搜索范围要覆盖全部源码文件 | E006 |
| 7 | **不要假设"这个结构体肯定有 err 字段"**。v1.5.5 的所有 struct 都没有 err 字段，v2.2.0 的全部都有。使用时必须本地定义或确认 | Batch 1 |
| 8 | **一次只读一个函数**。从目标版本中摘出一个函数，追踪其所有依赖，确认基线版本中对应的部分，再读下一个。不要同时读多个函数避免信息混乱 | 实操经验 |

### 2. 全链路追踪
- [ ] 追踪每条代码路径：用户态命令 → syscall → dispatch 入口 → 功能函数 → VFS 钩子
- [ ] 确认头文件 `#include` 链完整（每个用到的宏/结构体在哪个头文件定义）
- [ ] 确认函数签名是否变化：基线版本签名 vs 目标版本签名
- [ ] **确认注入标记在目标文件中确实存在**（注释标记、锚点行、`#endif` 等）
- [ ] **注入脚本必须检查插入是否成功，失败必须返回 False 终止构建**（切勿静默跳过）
- [ ] **功能可行性调研**：对每个新功能，逐项检查所有依赖在目标环境是否可用。依赖类型包括：全局变量、锁机制、外部函数、内核头文件宏。搜索范围覆盖项目全部源码文件

### 3. 边界条件和副作用验证
- [ ] 符号冲突检查：新函数名/全局变量名是否与其他模块重复
- [ ] 内核版本兼容性检查：所用内核 API 在目标内核版本是否可用
- [ ] 多次执行去重保护（幂等性）：同一脚本运行两次不应破坏代码
- [ ] `#ifdef` 条件完备性：新增 Kconfig 选项 = n 时，相关代码应完全跳过
- [ ] `__user` 注解正确性：所有用户态指针传递路径有 `__user` 标注
- [ ] 内存安全：`copy_from_user`/`copy_to_user` 返回值被检查，`kmalloc` 返回值被检查
- [ ] **脚本注入点验证：插入的结构体和函数声明应放在一起（避免半插入状态）**
- [ ] **缩进一致性：同一个 inject 脚本内不得混用空格和 tab。Python 字符串模板中 C 代码用 `\t` 转义，Python 本身用 4 空格缩进。替换 edit 后检查上下文缩进层级一致**
- [ ] **局部结构体自包含性：注入代码中本地定义的 struct 不得引用任何外部类型作为字段。所有字段必须平铺展开（基本类型或标准内核类型如 `struct list_head`），确保不依赖 `susfs.h` 或其他模块头文件中的自定义类型**

### 3b. ⚠️ 常见失败模式（从历史 bug 总结，逐条对照）
- [ ] 🔴 **静默跳过**：注入脚本的锚点 `if xxx in line` 不匹配时，是否返回 False 而不是无声继续？
- [ ] 🔴 **文件版本差异**：本地读的版本可能 ≠ CI 构建用的版本。不要假设本地文件等价于上游源文件
- [ ] 🔴 **"前向声明"错误**：编译报 `incomplete type / forward declaration` = 结构体定义没插进去。原因通常是锚点不匹配或插入点在 `#ifdef` 块外
- [ ] 🔴 **替换丢失 `#define`**：字符串替换时，确保替换范围包含完整行（`#define FOO 1` 而非只替换 `FOO`）
- [ ] 🔴 **双 `#define`**：替换目标内容本身以 `#define` 开头时，目标字符串也需以 `#define` 匹配
- [ ] 🔴 **CI 步骤顺序**：inject 脚本执行时，被修改的文件必须已存在。确认步骤顺序且不被后续步骤覆盖

### 4. 移植后全面审计（代码写完后、提交前）
- [ ] **Python 语法检查：`python3 -c \"import py_compile; py_compile.compile('scripts/xxx.py', doraise=True)\"`，每个修改过的 inject 脚本都要检查**
- [ ] 逐行审查每个改动文件，确认无笔误（变量名、函数名、拼写）
- [ ] 结构体布局与目标版本一致（注意 `err` 字段、新加字段的 ABI 兼容性）
- [ ] dispatch/入口条目（IOCTL + reboot/其他通道）都添加了
- [ ] **Kconfig 一致性：新增 `CONFIG_*` 时，`kernel-patches/*.config` 和 CI workflow 的 Kconfig 注册同步添加**
- [ ] 桩函数/兼容性代码更新/删除同步
- [ ] 去重保护代码与注入逻辑一致（避免半插入状态）
- [ ] **路径检查：确保无本地硬编码绝对路径**

### 5. 提交规范
- [ ] **提交前运行 `bash scripts/pre-flight-check.sh`，全部通过才允许提交**
- [ ] 提交信息格式：`feat: 项目名-版本 移植 - 功能说明`
- [ ] 提交信息包含移植依据、关键决策说明
- [ ] 只提交移植相关文件，不包括临时文件、备份文件

## 🔄 事后复盘：错误经验提炼（每次修复后立即执行）

每次解决一个 bug 后，必须在 [ERRORS.md](ERRORS.md) 新增条目。步骤如下：

### 步骤
1. 在 `ERRORS.md` 新开一个 `### E00N：` 条目
2. 记录四项内容：
   - **现象**：编译/运行时观察到什么
   - **根因**：为什么发生
   - **教训**：如何避免（具体操作，而非笼统建议）
   - **检查清单锚点**：关联到本文第 2/3/4 节具体哪条检查项
3. 如果新教训无法对应到现有检查项，则在对应节新增一条检查项
4. 如果适用于自动化检查，更新 `scripts/pre-flight-check.sh`
5. **⚠️ 经验提炼必须在下次提交前完成，不可跳过。`pre-flight-check.sh` 会自动检查：注入脚本有改动但 ERRORS.md 未更新 → 阻断提交**

### 判断标准
以下情况必须记录：
- 🔴 编译失败（任何原因）
- 🔴 运行时 crash（panic/Oops/重启）
- 🟡 功能不符合预期（命令输出错误、特征不起作用）
- 🟡 审计发现的结构体/签名不匹配
- 🟢 值得记录的心得（新的内核版本兼容性发现、工具用法技巧）

### 项目标识

| 版本 | 调用约定 | 返回类型 | 参数 |
